⚠ Verdict: Needs changes before next release. Codebase is structurally healthy and well-tested overall, but four issues — a WebSocket listener leak, a singleton-logger context bug, a userId/uniqueId conflation that can send wrong data to CS API, and a fail-soft pattern that makes CS errors invisible to monitoring — should be fixed before merging this branch to release/0.1.0.

Must Fix

4 issues
Severity Module Location Issue Suggested Fix
Critical websocket aws-websocket-client.ts:51-85 connectToAWS() is called on every reconnect, creating a new WebSocket and attaching three listeners (open, close, error) without removing them from the previous instance. The old WS sticks around long enough for its close_A handler to fire after this.ws = wsB, where it mutates shared state (reconnectAttempts, reconnectTimer) and the error_A handler closes the wrong socket via this.ws.close(). Causes cascading reconnects and orphaned socket allocations. At the top of connectToAWS(), call this.ws.removeAllListeners() and close the old socket if still open/connecting before creating a new one. Also clear this.reconnectTimer to prevent the race where sendToAWS() initiates a connect while a backoff timer is still pending.
Critical logger logger.service.ts:80-91 AppLogger is @Injectable() with default singleton scope. setContext() mutates a shared this.context field, so every controller/service that calls setContext(MyClass.name) overwrites the previous value. The last constructor to run wins, and every subsequent log line — across all services — is stamped with that context. Confirmed with the GET /maintenance/access log line showing context: "LicenseController". Mark AppLogger as Scope.TRANSIENT so each consumer gets its own instance. One-line change: @Injectable({ scope: Scope.TRANSIENT }). No call-site updates required.
Critical dispatch dispatch.service.ts:725-729 In the existing-user flow, userId is resolved as tokenPayload.userId ?? tokenPayload.uniqueId ?? null. uniqueId is the unauthenticated identifier used in self-registration (typically an email). If a token is missing userId but has uniqueId, the email is sent as the userId header to the CS API — wrong data, possibly failing CS validation. Drop the fallback. Existing-user tokens always carry userId after launch — if they don't, that's a contract violation, not data to coerce. Use userId: tokenPayload.userId ?? null and let the downstream null check return incomplete.
Critical dispatch dispatch.service.ts:776-781 Any non-5xx error from the CS API or the DB silently maps to { courseStatus: 'incomplete' } with HTTP 200. Datadog parent span stays green, the only signal is a warn-level log line that most alerting pipelines ignore. A user who actually completed the course gets reported as incomplete during transient failures — silent data corruption, invisible to monitoring. Bump the log to error level so log-based monitors fire. Tag the active dd-trace span as errored: tracer.scope().active()?.setTag('error', true). Optionally distinguish 4xx-not-found (legit incomplete) from 4xx-auth/timeout (real failure) and propagate the latter as 5xx.

Security Findings

2 issues
Severity Module Location Issue Suggested Fix
Security main main.ts:27 expressApp.set('trust proxy', true) trusts any upstream and accepts X-Forwarded-For from direct clients. req.ip is therefore spoofable, which can let an attacker forge a VPN-allowlisted IP to bypass the maintenance-mode VPN check, and to evade per-IP rate limits by rotating forged IPs. Compare with core-cdp-api's TrustProxies.php which lists explicit cluster CIDRs. Switch to env-specific CIDR allowlists matching the K8s cluster networks (e.g. ['10.10.0.0/16', '10.100.0.0/16', ...]). At minimum, set 'trust proxy', 1 if the ingress is the only proxy hop. Acknowledged as deferred by team — track as a hardening task.
Security throttler .env.example:76 — app.module.ts:42-56 THROTTLE_ENABLED=false in the example env disables the global rate limiter via skipIf. If prod env vars inherit the example default and don't explicitly set true, the API is unprotected from per-IP request floods (including auth flooding on /auth/tokens which does bcrypt 12-rounds). Verify deployed environments set THROTTLE_ENABLED=true. Consider flipping the default to true and require explicit opt-out for local dev. Add a tighter named throttler for auth routes (e.g. 30/min/IP) to protect against credential stuffing on bcrypt-bound endpoints.

Should Fix

10 issues
Severity Module Location Issue Suggested Fix
Performance axios axios.service.ts:9 axios.create({ timeout: 30000 }) uses default HTTP/HTTPS agents with no keepAlive. Every outbound call to RQI/CS/MS opens a fresh TCP+TLS connection (~50–100ms wasted per call), and sockets pile up in TIME_WAIT — on a busy pod, ephemeral ports can exhaust. Add httpAgent: new HttpAgent({ keepAlive: true, maxSockets: 50 }) and the equivalent httpsAgent. Reuses connections and dramatically reduces socket churn.
Lifecycle cache cache.module.ts CacheModule creates a ioredis client via useFactory but has no onModuleDestroy to call client.quit(). On SIGTERM, the Redis socket lingers until the process exits — K8s may force-kill the pod past the grace period, and any in-flight pipelines aren't drained cleanly. Add a small @Injectable() ValkeyDisposer implements OnModuleDestroy provider that injects 'VALKEY_CLIENT' and calls await client.quit() in onModuleDestroy.
Correctness announcements announcements.controller.ts:25-32 Endpoint takes @CurrentUser() user: JwtUserPayload but only uses user.userId for a log line. The actual response is identical for every authenticated user — globally fetched announcements filtered to student role + LMS org type 102. Cache key is global. Docstring claims "active announcements for the authenticated user" which is misleading. Confirm with stakeholders whether org-scoped filtering is intended. If yes, include user.organizationId in cache key and read-time filter. If not, drop @CurrentUser() and fix the docstring to describe the IP/role-only behavior.
Resilience announcements announcements.service.ts:383 fetchAnnouncementsWithLimitedRetry uses a bare setTimeout for backoff — not stored in this.retryTimeoutId, so it can't be cancelled on shutdown. Inconsistent with fetchAnnouncementsWithRetry which does register the timeout. In-flight user-facing requests block module destruction for up to ~7s on shutdown. Store the timeout in this.retryTimeoutId, clear it in onModuleDestroy, and check this.isShuttingDown after each await — same pattern already used by the background job retry.
Resilience license license.service.ts:372 Same pattern: await new Promise((resolve) => setTimeout(resolve, 500 * attempt)) inside a CS PUT retry loop. Bare setTimeout, unstoppable on shutdown. Up to ~7.5s of hang per in-flight request during graceful shutdown. Wrap with cancellation token similar to announcements.service.ts pattern, or accept the short delay as bounded and document it.
Correctness dispatch dispatch.service.ts:760 const cacheKey = `course-status:${licenseId}`; — cache key omits userId. Today licenses are user-scoped so this is safe, but any future flow that shares licenses across users (proxy launches, vendor pooling) would leak completion status between users. Include userId in the key for defensive isolation: course-status:${licenseId}:${userId}.
Brittleness announcements announcements.service.ts:441 org_type.id === 102 — magic number for "learner" org type. If RQI seeds change between envs (different IDs in dev vs prod) or RQI changes the ID, this filter silently returns 0 announcements with no error. Match by org_type.code or org_type.name (string), or move the ID to env config (e.g. RQI_LMS_ORG_TYPE_ID) so it can be tuned per environment without redeploy.
Brittleness announcements announcements.service.ts:103-117 new Date(announcement.starts_at.replace(' ', 'T') + 'Z') appends Z regardless of what RQI sends. If RQI starts including offsets (2026-05-04 12:00:00+05:30), this produces an invalid date string and silently filters out all announcements. Validate the input shape with a regex first. If the format changes, fail loudly with an error log rather than silently dropping data.
Inefficiency content-service content.service.ts:218 tcId = this.configService.get<string>('app.csToolConsumerGuid') is re-read from config on every fetchCourseStatusFromCsApi call. The constructor at line 41 already validates it exists. Wasted lookups, and TS type is string | undefined which leaks into the headers object. Store as a private field this.csToolConsumerGuid: string set in the constructor — same pattern as this.csApiCompletionStatusUrl already uses.
Type Safety auth auth.service.ts:382-388 Proxy refresh path uses const tokenPayload: any = { ... } and conditionally adds source/proxiedBy. Same pattern repeats for meta: any. Loses TS coverage on a security-relevant field (proxy provenance) — a typo or wrong property name would compile silently. Use a discriminated union or extend UserAuthTokenPayload with optional source?/proxiedBy? fields, then build the object with proper types.
