⚠ Verdict: Needs changes before merge. The codebase is well-structured and broadly well-tested, but several blocking issues remain: a hardcoded fallback JWT signing secret, a shared-singleton logger that misattributes every log line, two check-then-act races (one-time launch-token replay and duplicate license creation), a missing-await that races cache invalidation, an unguarded version cast that can crash a launch, and a completion callback with no idempotency that re-issues eCards. The recurrence feature this branch is named for is currently shipped as dead code behind const false flags.

Must Fix

6 issues
Severity Module Location Issue Suggested Fix
Critical logger logger.service.ts:83,89-91 AppLogger is a @Global() default-scoped singleton with a mutable private context field. ~25 services call this.logger.setContext(X.name) in their constructors against the same instance, so whichever provider is constructed last wins and formatLog stamps every subsequent log line with that context. Result: log lines are attributed to the wrong module, plus a shared-state write race. Stop storing context on the shared instance. Either mark AppLogger as Scope.TRANSIENT so each consumer gets its own instance, or use Pino child loggers (logger.child({ context })) and require callers to pass context per call.
Critical rqillp-service rqillp.service.ts:451-461 updateUserCustomProfile does not await the rqiRequest PATCH, and fires the two cacheService.delete(...) calls without awaiting them. A concurrent getUserById can repopulate the cache with stale data after the delete, and the returned promise resolves before the PATCH completes. It also only busts two hard-coded keys while getUserById caches under rqillp:user:${userId}${:filter} for arbitrary filters, leaving other variants stale. await the request, then await the deletes sequenced after a successful PATCH. Invalidate by key pattern (e.g. SCAN rqillp:user:${userId}*) or normalize to a single cache key.
Critical dispatch dispatch.service.ts:601-637 version: courseVersion as string where courseVersion is undefined whenever the course has no manifest. The Prisma version column is a required String with no default, so the create path attempts to persist undefined and throws a Prisma validation error that surfaces as an unhandled 500 mid-launch. The as string cast merely hides the type hole. Handle a missing manifest version explicitly — fail fast with a clear error or default to a sentinel/empty string — rather than casting undefined to string.
Critical license / LIS lis-completion.service.ts:233-253 processLisCompletion never reads the license completion status. The route and service docstrings claim idempotency ("Already Processed" when already complete), but every duplicate completion/course callback re-runs triggerPostCompletionActions → processCertificateClaim → msService.createCertificate. Duplicate Content-Service callbacks therefore re-issue eCards. Check the license status (or an already-completed marker) before claiming, mark it completed in the DB, and return "Already Processed" on repeats. If idempotency is delegated to MS, remove the false claim and document it.
Critical repositories user-license.repository.ts:88-108 getOrCreateUserLicense is a check-then-create over a non-unique index (@@index([user_id, org_id, course_id])). Two concurrent POST /dispatch/auth calls for the same user/course both miss the findFirst and both create, producing duplicate activated licenses. This is the primary licensing path for launch and self-register, and callers do not tolerate duplicates. Add a unique constraint on the active tuple and use upsert/catch P2002, or wrap select+insert in a transaction with appropriate isolation.
Critical auth auth.service.ts:504-510 In exchangeExternalServiceToken, userToken can be undefined (getTokens types it optional and getOrCreateUserToken may return empty), yet it is non-null-asserted (userToken!) and returned. The client receives accessToken: undefined in a successful 200 response instead of a surfaced failure. Guard before constructing the response: if (!userToken) throw new ServiceUnavailableException(...).

Security Findings

10 issues
Severity Module Location Issue Suggested Fix
Security auth jwt.strategy.ts:19 secretOrKey: configService.get<string>('auth.jwt.secret') ?? 'your-secret-key'. If the secret env var is ever unset/empty, the strategy silently validates tokens against a public, well-known fallback key, allowing forged JWTs. token.service.ts already throws on a missing secret, so this fallback is both insecure and inconsistent. Remove the ?? 'your-secret-key' fallback and fail fast (throw at startup) when the secret is absent. Never provide a default signing/verification key.
Security main main.ts:27 expressApp.set('trust proxy', true) blanket-trusts every upstream hop, making req.ip / X-Forwarded-For fully spoofable. This undermines the per-IP ThrottlerGuard on public auth endpoints and poisons any IP-based logic (e.g. the maintenance VPN allowlist). Set trust proxy to the specific number of known proxy hops or a CIDR allowlist of the load balancer / ingress — never blanket true.
Security main main.ts (whole file) No Helmet / security headers anywhere; helmet is not even a dependency. The app sends no X-Content-Type-Options, Strict-Transport-Security, X-Frame-Options, etc. Add helmet and app.use(helmet(...)) early in bootstrap(), tuned for the LTI iframe-launch flows that need framing.
Security auth token.service.ts:534-551 verifyLaunchToken single-use enforcement is a check-then-act race: exists() then set() are not atomic. Two concurrent requests with the same launch/proxy one-time token both pass the exists check before either writes the "used" marker, so an OTT can be redeemed more than once. Use an atomic set-if-not-exists (SET key val NX EX ttl) and treat a falsy result as "already used"; add that primitive to CacheService.
Security curriculum curriculum.service.ts:259-261 computeRetakeEligibility() is a stub that unconditionally returns true and is exposed as retakeEligible in the live course list. Combined with createRetakeLicense (which only checks that any non-deleted license exists, not actual completion/eligibility), any learner with a license can mint unlimited retake licenses. Implement the documented real check (product course_retake/retake_duration vs completion timestamp) before shipping, or gate the retake endpoint behind a feature flag; ensure createRetakeLicense enforces real eligibility.
Security auth auth.service.ts:137-138,256-316 The proxy-token path mints a full user session from userId/organizationId/courseId/licenseId taken straight from the proxy JWT, with no DB validation that the license still exists, belongs to that user/org, or is active. A proxy token issued before a license was revoked/expired still yields a live session. Validate the license is active and that userId/org match the license at exchange time, or keep proxy-token TTL extremely short and document the trade-off explicitly.
Security auth token.service.ts:637-700 Refresh-token rotation stores only the latest jti keyed by refresh:{userId} (single slot), so concurrent sessions on multiple devices clobber each other and a legitimate retry trips the "reuse detected" force-logout. Cache failures are also swallowed silently, degrading reuse detection to "no protection" when Valkey is unavailable. Key the jti store per-session (include a device/session id), and decide explicitly whether cache unavailability should fail open or closed for refresh.
Security database prisma.service.ts:38-40 TLS is configured with rejectUnauthorized: true but checkServerIdentity: () => undefined, which disables hostname verification. This is only safe with a pinned CA, yet tlsCaFile is optional — with SSL on and no CA file you get the system CA bundle with hostname checks off, enabling MITM. Only override checkServerIdentity when a private CA file is actually loaded; otherwise keep default hostname verification.
Security dispatch dispatch.service.ts:232 validateVendor builds the cache key as vendor:${vendorId}:${vendorKey}, embedding the vendor secret key in a Valkey key. Cache keys are commonly visible via KEYS/SCAN/monitoring output, leaking the credential. Hash the vendorKey (e.g. SHA-256) into the key, or key only on vendorId with the key checked separately.
Security throttler / config throttle.config.ts:4 enabled: process.env.THROTTLE_ENABLED === 'true' means anything other than the literal 'true' (including unset) disables rate limiting entirely via skipIf. Combined with the spoofable trust-proxy setting, deployed envs may run with no working rate limit on bcrypt-bound auth endpoints. Default to enabled (!== 'false'), ensure deployed configs set THROTTLE_ENABLED=true, and add a tighter named throttler for auth routes.

Maintainability dispatch dispatch.service.ts:578-638 Large dead branch: createNewLicense and usePreviousLicense are hardcoded const false, so the recurrence blocks are unreachable. findSecondLatestActivatedLicense, validateRecurrence, fetchAndValidateCourseStatusForRecurrence, and toSystemDateString have no production callers, and their tests are describe.skip. This is the feature this branch is named for. Either wire the recurrence path back in (with the skipped tests re-enabled) or remove the ~250 lines of unreachable code before merge.
Performance axios axios.service.ts:9-11 No HTTP/HTTPS keep-alive agent on the shared axios instance. The entire integration layer (rqillp, MS gateway, content service, token endpoints) routes through this instance, so every outbound call pays a fresh TCP + TLS handshake and sockets churn in TIME_WAIT. Configure httpAgent/httpsAgent with new Agent({ keepAlive: true, maxSockets }).
Performance auth / integrations external-auth.service.ts:53-141 No cache-stampede / single-flight protection on token fetches. On a cold cache or simultaneous expiry, N concurrent requests all miss and all hit the token endpoint at once (thundering herd against rqillp, MS, and content-service auth). Coalesce in-flight token fetches via a per-key promise map (single-flight), or add a short lock with jitter.
Correctness auth / integrations external-auth.service.ts:164-220 getOrCreateGatewayAuthToken caches for ONE_DAY - 300s regardless of the gateway token's actual lifetime (the response token is never introspected for expiry). If the gateway issues shorter-lived tokens, this caches stale/expired tokens for up to a day. Derive TTL from the token's real expiry (decode JWT exp or use a returned expiry field), as computeCacheTtl already does for the other tokens.
Lifecycle websocket aws-websocket-client.ts:105-108 sendToAWS calls connectToAWS() directly when the socket is CLOSED, with no check for an in-flight reconnectTimer. A pending backoff reconnect plus a sendToAWS-triggered reconnect can spawn two concurrent connections; the orphaned socket keeps its old listeners. Clear any existing reconnectTimer and removeAllListeners() + close the previous socket before reconnecting; guard concurrent connects with an isConnecting flag.
Security websocket aws-websocket-client.ts:49,100 Full WebSocket URL (including the userId query param) is logged at info, and every outbound payload is logged in full. If payloads carry PII or the URL embeds a signed token, this leaks to logs. Log host/path only, redact the query string, and avoid logging full payloads at info.
Correctness curriculum curriculum.service.ts:618-619 addMonths(validFrom, courseData.course_validity) assumes course_validity is a valid number. If upstream omits/nulls it, addMonths(date, undefined) yields an Invalid Date that gets persisted as valid_to. Validate course_validity is a finite positive number before calling addMonths; throw a clear error otherwise.
Correctness dispatch dispatch.service.ts:154-171 validateProduct dereferences product.course.id/.status without guarding product.course. It is typed non-null but comes from an external API; if omitted, this throws a TypeError caught by a bare rethrow and surfaces as a 500 rather than a clean validation failure. Null-check product.course and return vdr_not_sub.
Code Quality dispatch dispatch.service.ts:195-197,292-294,450-452 Three catch (error) { throw error; } blocks add nothing — no logging, no normalization. Dead try/catch. Remove them, or add the intended logging/normalization.
Code Quality auth auth.service.ts:82-91 validateUser is a no-op that spreads the payload and stamps role: 'user'; the comment "check if user exists in database" is stale/misleading and nothing is validated. A revoked user with a still-valid access token stays authenticated. Implement real validation (existence/active check, ideally cached) or remove the misleading comments and rename to reflect it is a pass-through enrichment.
Type Safety auth auth.service.ts:196,221,370,378 Heavy any usage in a security-critical module: createUnregisteredUserResponse(vendorDetails: any), meta: any, and prepareAuthResponse(result: any) returning responseBody: any. Payload shape matters here. Type vendorDetails as the launch/registration token payload union and result as the authenticate return type.
Correctness common / http http-error.handler.ts:21 Upstream 403 Forbidden is mapped to UnauthorizedException (401), losing the authorization-vs-authentication distinction and surfacing a misleading 401 to clients. Map 403 to ForbiddenException.
Security curriculum course-launch-request.dto.ts:12-15 licenseId is validated only with @IsString(), whereas CourseListRequestDto.licenseId correctly uses @IsUUID('4'). The value is looked up as a license that feeds OAuth signing; a malformed id hits Prisma and may throw an unmapped error instead of a clean 400. Use @IsUUID('4') for consistency and early rejection.
Config config app.config.ts:8 corsOrigins: [] is hardcoded to an empty array and never reads an env var, so the strict CORS allowlist branch in main.ts always resolves to origin: false — either dead config or cross-origin cookie-auth endpoints are entirely blocked. Populate from process.env.CORS_ORIGINS (comma-split) or remove the unused allowlist branch.
Config config app.config.ts:11 rqillpServiceUrl defaults to a hardcoded dev URL. A missing env var silently routes core LMS traffic to a dev host in any environment. Remove the default; fail fast if unset in non-local envs.
Type Safety certificates certificates.service.ts:273,286-288 fetchPromises: Promise<any>[] with positional indexing (fetchResults[0]?.data, [1], [2]) is fragile: the eCard fetch is conditionally pushed, so indices depend on flag state. Build the promises with explicit named variables (await a typed tuple) instead of positional any[].
