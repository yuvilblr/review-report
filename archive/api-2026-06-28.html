⚠ Verdict: Needs changes before release. The codebase is well-structured and broadly well-tested, but several blocking issues remain: a hardcoded fallback JWT signing secret, a non-idempotent LIS completion callback that re-issues eCards, a Redis client never closed on shutdown, a missing-await that races cache invalidation, a one-time launch-token replay race, and a CORS allowlist hardcoded empty so cross-origin cookie-auth silently fails.

Must Fix

5 issues
Severity Module Location Issue Suggested Fix
Critical license / LIS lis-completion.service.ts:233-253
license.controller.ts:78		 The LIS completion endpoint is not idempotent, contradicting its own contract. The controller docstring promises an "Already Processed" response when already complete, but processLisCompletion performs no completion-state check — every completion/course callback re-fires the WebSocket notification and re-invokes processCertificateClaim → msService.createCertificate. Content Service legitimately retries callbacks, so duplicate eCards/certificates result. Gate triggerPostCompletionActions behind a persisted completion flag (status/timestamp, or dedupe on attemptId) and return "Already Processed" on repeats.
Critical cache cache.module.ts:13-25 The VALKEY_CLIENT ioredis provider is a raw factory with no onModuleDestroy/quit(). main.ts calls enableShutdownHooks(), but nothing closes the Redis connection — the socket and its keepAlive timer are orphaned on every restart/redeploy. Wrap the client in a provider implementing onModuleDestroy that await client.quit().
Critical rqillp-service rqillp.service.ts:448-458 updateUserCustomProfile doesn't await the PATCH before invalidating the cache, and the two cacheService.delete(...) calls are floating (un-awaited). The deletes race the write — a concurrent getUserById can re-cache stale data after the delete but before the PATCH commits — and only two hardcoded keys are busted (other filter variants stay stale). await the request, then await the deletes; invalidate by key pattern (SCAN rqillp:user:${userId}*) or normalize to one key.
Critical config / CORS config/app.config.ts:8 corsOrigins is hardcoded to [] and never reads process.env.CORS_ORIGINS (which is documented in .env.example). So main.ts always evaluates the allowlist to false and every non-/dispatch/ cross-origin request is blocked regardless of deployment config — any cookie-auth browser client on a different origin silently fails CORS. Parse the env var: (process.env.CORS_ORIGINS ?? '').split(',').map(s => s.trim()).filter(Boolean).
Critical auth token.service.ts:531-553 verifyLaunchToken enforces single-use with a check-then-act race: it calls cacheService.exists() then later cacheService.set(). Two concurrent requests with the same launch/proxy token both pass the exists check before either writes the marker, so a one-time token (which mints a full access+refresh session) can be redeemed more than once. Use an atomic set-if-not-exists (SET key val NX EX ttl) and treat a falsy result as "already used"; add a setNx primitive to CacheService.

Security Findings

10 issues
Severity Module Location Issue Suggested Fix
Security auth jwt.strategy.ts:19 secretOrKey: configService.get<string>('auth.jwt.secret') ?? 'your-secret-key'. If JWT_SECRET is unset, the strategy validates tokens against a publicly-known key, allowing forged tokens — a fail-open posture (auth.config.ts leaves the secret undefined when the env is missing). Remove the fallback and fail fast at boot if auth.jwt.secret is absent.
Security bootstrap main.ts:27 expressApp.set('trust proxy', true) trusts the entire X-Forwarded-For chain, making req.ip spoofable. This feeds the ThrottlerGuard and the maintenance VPN/IP allowlist — a client can forge their source IP to bypass per-IP rate limiting (and potentially the VPN gate). Set a bounded value — a trusted proxy CIDR list or hop count matching the real ingress topology.
Security bootstrap main.ts (whole file) No Helmet / security-response-headers middleware anywhere (helmet isn't a dependency). Responses ship without X-Content-Type-Options, X-Frame-Options/CSP, Referrer-Policy, HSTS, etc. Add helmet() in bootstrap(), tuned for the LTI iframe flows.
Security config / throttle config/throttle.config.ts:4 enabled: process.env.THROTTLE_ENABLED === 'true' defaults to false when unset, and .env.example ships THROTTLE_ENABLED=false. The configService.get('throttle.enabled', true) default is dead (the key always exists). Net: rate limiting is OFF unless explicitly enabled in every deployed env — a fail-open default for a security control. Default to enabled (!== 'false') and set THROTTLE_ENABLED=true in deployed configs.
Security dispatch dispatch.service.ts:215 The Valkey cache key embeds the raw vendor secret: vendor:${vendorId}:${vendorKey}. Secrets in cache keys leak into KEYS/SCAN output, the Redis slow-log, and key-dump tooling. Hash the key (vendor:${vendorId}:${sha256(vendorKey)}) or key only on vendorId and compare the key in the payload.
Security auth token.service.ts + auth.module.ts:27-31 JWTs are signed/verified with no issuer/audience binding (verify uses default options). All token classes (launch, registration, dispatch, proxy, access, refresh, app) share one secret, distinguished only by a type claim — cross-context replay is prevented solely by per-method string checks. Add verifyOptions: { issuer } (and per-type audience) so tokens are cryptographically bound, not just string-claim-checked.
Security auth auth.controller.ts:330-337 POST /auth/tokens/exchange is reachable by app-role tokens (neither @AppOnly() nor user-restricted). For an app principal user.userId is undefined, so the service calls the external API with /users/undefined/tokens. This is a user-only token-exchange endpoint. Restrict to user principals (reject role==='app' / validate isUserAuthTokenPayload) and reject when userId is missing.
Security curriculum dto/course-launch-request.dto.ts:12-15 licenseId is validated only with @IsString(), while CourseListRequestDto uses @IsUUID('4'). The value reaches findOne({ id }) against a UUID PK unfiltered. The ownership check still prevents IDOR, so this is defense-in-depth. Use @IsUUID('4') for consistency and edge rejection.
Security config / auth config/auth.config.ts:36-39 Refresh-token cookie sameSite defaults to 'none' when COOKIE_SAME_SITE is unset, and secure is forced only when NODE_ENV==='production'. A non-prod deployment could emit a SameSite=None cookie without Secure (invalid / dropped, or broadly cross-site exposed). Default sameSite to 'lax' and/or force secure: true outside local dev.
Security websocket / content aws-websocket-client.ts:78,136
content.service.ts:256		 Full outbound WebSocket payloads and the connection URL (incl. userId) are logged at info level, and the full CS API completion response body is logged on every call. These carry user/license identifiers and aren't covered by the request-logging redaction. Drop payloads/bodies from info logs (or log non-PII metadata at debug); strip the query string from the URL log.

Should Fix

16 issues
Severity Module Location Issue Suggested Fix
Lifecycle logger logger.service.ts:89-91 AppLogger is a shared singleton, but several services call setContext(...) in their constructors. Since it's one instance, the last service to initialize wins, and any log() without an explicit context is mislabeled. Make the logger transient-scoped, or drop setContext and always pass context per call.
Performance axios axios.service.ts:9-42 The shared axios instance has no httpAgent/httpsAgent with keepAlive, so every outbound call to rqillp/MS/content opens a fresh TCP+TLS connection. The request/response interceptors are also pure no-ops (dead commented auth code). Configure keepAlive: true agents; remove or implement the interceptors.
Performance auth / integrations external-auth.service.ts:57-290 No request coalescing on any token-fetch path — on a cold cache, N concurrent requests each fire a real /tokens call (cache stampede), hammering the auth endpoint. Coalesce in-flight fetches via a per-key promise map cleared on settle.
Correctness common / http http-error.handler.ts:20-29 Upstream 403 is mapped to UnauthorizedException (401) and 500 to ServiceUnavailableException (503), silently rewriting upstream status and losing the authz-vs-authn distinction. Map 403 → ForbiddenException, 500 → InternalServerErrorException/BadGatewayException.
Correctness dispatch dispatch.service.ts:566-589 Check-then-act race in getOrCreateUserLicense: two concurrent calls for the same (user_id, org_id, course_id) both miss the findFirst and both create; the partial unique index makes the second throw P2002, which is not caught and propagates as an unhandled 500. Catch P2002 in getOrCreateUserLicense and re-query/return the existing row.
Resilience dispatch dispatch.service.ts:719-736 In the existing-user branch of resolveLicenseAndUserId, findOne is not wrapped in try/catch (unlike the self-reg branch). A transient DB error throws a 500 instead of degrading to incomplete, inconsistent with the documented graceful-degradation behavior. Wrap the existing-user lookup in the same try/catch returning null values.
Quality dispatch dispatch.service.ts:484-496 getOrgLanguage(dto.vendorId) runs in Promise.all alongside validateVendor with the unvalidated vendorId, so it fires an external call and populates the org-language cache for arbitrary attacker-supplied org IDs even when the vendor is invalid. Fetch language only after vendor validation succeeds.
Error Handling auth auth.service.ts:498-519 exchangeExternalServiceToken swallows the underlying error with a bare catch {} and always returns ServiceUnavailableException (even for 401/invalid-credential failures); userToken! is non-null asserted though typed optional. Inspect the caught error and map auth failures to UnauthorizedException; guard against a missing userToken.
Code Quality auth auth.service.ts:82-91 validateUser is a no-op scaffold (spreads the payload, stamps role:'user') with a stale "add your custom validation logic" comment; the if (!user) guard in the strategy can never be true. Remove the dead branch/comment, or implement real validation.
Type Safety auth auth.service.ts:138,196,221,370,378 Heavy any in a security-critical module (prepareAuthResponse(result: any), vendorDetails: any, meta: any); casts at 138/151 bypass the structural type guards. Narrow with the existing type guards and replace any with concrete payload types.
Correctness curriculum dto/course-launch-request.dto.ts:51-66 customDueDate, customStartDate, customCourseName are declared (the latter two @IsNotEmpty() required) but launchCourse() never reads them — clients are forced to send mandatory fields that are silently ignored. Consume them in the LTI payload, or remove them (at least make optional if kept for forward-compat).
Type Safety curriculum curriculum.service.ts:223 The date-mutation loop casts each course to any and mutates response.courses[category] in place via untyped indexing, bypassing the CourseListResponseDto contract. Type the course category arrays on the response DTO and iterate typed objects.
Config config config/app.config.ts:9-11 rqillpServiceUrl falls back to a hardcoded dev URL when RQILLP_SERVICE_BASEURL is unset — a misconfigured prod deploy would silently point the core LMS integration at the dev host. Require the env var (throw on missing) rather than defaulting to a dev host.
Security bootstrap main.ts:39-46 The global text/plain → urlencoded re-parse runs on every request whose content-type merely includes('text/plain'), with no body-size guard beyond Express defaults and a fragile leading-& strip; req.body is reassigned mid-pipeline. Scope this to the LTI/dispatch callback routes rather than applying globally, and add a size limit.
Config config / auth config/auth.config.ts:8-12 JWT lifetimes (expiresIn, refreshExpiresIn, launch/registration token TTLs) have no fallback and will be undefined if env vars are missing — behavior then depends on how the signing lib treats undefined (often "never expires"). Provide explicit safe defaults or validate presence at startup.
Robustness certificates certificates.service.ts:286-304 userDetails = fetchResults[0]?.data is optional-chained, but userDetails.full_name/first_name (and orgDetails) are then accessed without a guard; an empty RQI data throws a raw TypeError instead of a meaningful exception. Validate userDetails/orgDetails are present and throw a domain exception.

  • Existing-user DB-error degradation in resolveLicenseAndUserId untested
  • updateUserCustomProfile await-ordering / floating-delete race untested
  • Token-fetch stampede/coalescing untested (no coalescing exists)
  • No test that boot fails when JWT_SECRET is unset, nor that issuer/audience is verified
  • No test that an app-role token is rejected at /auth/tokens/exchange
  • Throttle enabled default and CORS corsOrigins wiring untested — a config spec would have caught both mis-defaults
  • license-expiration.job.spec.ts shallow — error branch (process.exit(1)), multi-batch loop, and empty-result break untested
  • Certificates partial-failure path and absent userDetails/orgDetails untested; main.ts bootstrap (trust-proxy, CORS, text/plain reparse) has no e2e assertions
