⚠ Verdict: Needs changes before release. The app is well-structured and unusually well-tested, but a cluster of security and lifecycle issues should be addressed first: a hand-rolled HTML sanitizer feeding dangerouslySetInnerHTML (DOMPurify isn't installed), wildcard postMessage('*') on course completion, an auth temp-token left in the URL, external refresh tokens held in JS memory, no Datadog URL scrubbing, plus a shared WebSocket that's never disconnected on unmount and silently sticks to a stale license on course switch.

Must Fix

4 issues
Severity Module Location Issue Suggested Fix
Critical course-socket course-socket.ts:27-35
courses-view.tsx:39
lti-launch-view.tsx:29		 connect() is called from useEffect with no cleanup, so the module-level WebSocket and its 30s keep-alive interval are never disconnect()-ed on unmount or route change. The connection outlives the component, and each reconnect's onopen registers a fresh keepAliveTimer — timers and sockets accumulate. Return () => disconnect() from the effects. Because two components share the singleton, use ref-counting or a single owner so one unmounting view doesn't tear down the other's socket.
Critical course-socket course-socket.ts:27-32,79-82 On connect(newLicenseId), currentLicenseId is reassigned but openSocket() early-returns when a socket is already OPEN/CONNECTING. The live socket stays subscribed to the old userId=<oldLicenseId>_0 while the app believes it's listening for the new license — completion notifications for the new course are silently missed. If licenseId !== currentLicenseId and a socket exists, disconnect() before opening the new one.
Critical routing / auth routes/courses/index.tsx:16-29 CoursesRoute reads session data from the in-memory Zustand store with no RequireAuth guard. On a hard refresh the store is empty, so a registered user is shown a 401 ErrorFallback instead of being routed back through /auth. The fallback is also rendered in the body (not thrown), so the route's errorElement is bypassed. Add a shared auth guard that redirects to the auth/launch flow (or throws a Response so the errorElement handles it) instead of inline-rendering the fallback.
Correctness routing app/router.tsx:94-98 The router.subscribe effect references router but has an empty dependency array. router is recreated by useMemo whenever queryClient changes, so a new router would never be re-subscribed — showInitLoader goes stale and the previous subscription leaks. Add router to the dependency array: useEffect(() => router.subscribe(...), [router]).

Security Findings

6 issues
Severity Module Location Issue Suggested Fix
Security courses course-info.tsx:35,98,105 sanitizeHtml is a hand-rolled DOM-stripping sanitizer (removes script/iframe + on* attrs) feeding two dangerouslySetInnerHTML sinks. It's bypassable: it doesn't strip javascript:/data: URLs in href/src, misses <svg>/<math> mXSS vectors, and temp.innerHTML = html can trigger side effects. The file's own comment says to use DOMPurify — which is not installed. Add DOMPurify and replace sanitizeHtml with DOMPurify.sanitize(html).
Security lti / socket courses-view.tsx:46
course-socket.ts:56		 window.opener?.postMessage('cdp_course_completed', '*') posts to a wildcard target origin — any window that opened (or framed) this app receives the completion event. Post to a specific configured parent origin (e.g. config.PARENT_ORIGIN), never '*'.
Security auth routes/auth/exchange-token.tsx:28-29 The temporary auth token arrives in the URL query string (/auth?token=...). Tokens in URLs leak via browser history, Referer headers, and proxy/server logs. On the error path the token-bearing URL persists in history and is rendered by the error boundary. Ensure post-exchange redirect uses replace semantics, redirect to a clean path on failure too, and confirm the token is short-TTL / single-use server-side.
Security external-api-client external-api-client.ts:18,29-39 External-service refreshTokens are held in the in-memory Zustand store and POSTed directly from the browser to refreshUrl. A refresh token in JS memory is XSS-exfiltratable, and this violates the project convention (LP refresh token is HTTP-only cookie) for the external-service tier. Keep external refresh tokens out of JS; refresh exclusively through the LP backend exchange (HTTP-only cookie) and drop the direct client-side refresh path.
Security datadog datadog/index.ts:17-29 datadogRum.init has no beforeSend scrubbing, with trackResources + trackUserInteractions on. defaultPrivacyLevel: 'mask-user-input' masks DOM input but not URLs — so the auth ?token= param and the socket userId=<licenseId> param are captured in RUM. Add a beforeSend that strips token/userId query params from event.resource.url and event.view.url.
Security lti lti-iframe.tsx:36 The LTI iframe sets allow="usb; serial; local-network-access; microphone *" — granting powerful hardware features (microphone * = all origins) — and has no sandbox attribute. Combined with auto-POSTing OAuth-signed params to a server-supplied lti_endpoint, the embedded content is broadly trusted. Scope allow to the specific content-service origin and only the features actually needed; validate lti_endpoint scheme/host against an allowlist before submit.

Should Fix

14 issues
Severity Module Location Issue Suggested Fix
Correctness api-client api-client.ts:49-53 getValidToken() fires a fire-and-forget proactive refresh (refreshAccessToken().catch(()=>{})) then returns the stale token anyway. A request made within the 60s buffer can go out with a token that expires mid-flight; the refresh only benefits the next request. await the refresh when expiring-soon and return the new token (make getValidToken async), or rely solely on the reactive 401 retry.
Resilience course-socket course-socket.ts:68 Reconnect uses setInterval(openSocket, RECONNECT_MS) with no exponential backoff and no max-retry cap. If the server keeps closing the socket, this hammers reconnects on a fixed cadence. Use a self-rescheduling setTimeout with exponential backoff and a retry ceiling.
Correctness auth-store use-auth-store.ts:87-97 getAccessToken() is a getter with a side effect: it calls set() to clear state when the token is expired. Invoked during render (via isAuthenticated()) it can trigger a state mutation mid-render and extra re-renders. Move expiry-clearing out of the read path (a separate pruneExpired() action), or compute expiry without mutating.
Correctness auth-store use-auth-store.ts:77-85 clearAuthData() clears tokens/user/externalTokens but not source, so a stale proxy source lingers after logout. Reset source: null in clearAuthData().
Correctness api-client api-client.ts:32-37 The /auth/refresh response is read as data.accessToken/data.expiresIn with no validation; a 200 with an unexpected body calls setAccessToken(undefined, undefined) and poisons the store. Validate the refresh response shape before calling setAccessToken.
Resilience external-api-client external-api-client.ts:30-39 The tier-2 refresh fetch has no try/catch around res.json() or the network call, so a thrown error escapes instead of falling through to the tier-3 backend exchange as the comment implies. Wrap in try/catch so failures fall through to the LP backend exchange.
Correctness courses courses-view.tsx:44-62 The all-completed effect reads t, toast, toastStyle but its dependency array is only [...allCompleted] — stale-closure risk; exhaustive-deps is silently violated. Include the referenced values in the dependency array (or memoize them).
Type Safety courses course-credential.tsx:40-42 certificateData/ecardData are typed any despite existing Certificate/CertificateData types — TS strict violation. Type these as Certificate | undefined.
Dead Code lti queries/use-launch-params.ts:9-14 Exported launchParamsQueryOptions builds queryOptions with no queryFn and is unused outside its own test — misleading/dead code that would throw if consumed. Remove it, or give it the real queryFn so it matches useLaunchParams.
Security error-fallback error-fallback.tsx:127 console.error('Error caught:', errorDetails) logs the full error object (incl. stack and data) to the browser console on every fallback render, in production. Gate the console output behind isDevelopment (as provider.tsx does).
Quality app shell main.tsx:14-16 <StrictMode> is commented out, so React's double-invocation checks that surface effect-cleanup bugs are lost — directly relevant to the router.subscribe and socket-cleanup issues above. Re-enable <StrictMode> and fix anything it surfaces.
UX layouts layout-wrapper.tsx:22-33 Route-level Suspense and the state === 'loading' branch both render a full-page loader, with a nested Suspense on Outlet using the same fallback — every navigation replaces the whole content region (flash of empty content / layout jump). Keep the prior Outlet mounted during loading transitions (overlay spinner) instead of unmounting it.
i18n app / layouts not-found.tsx:8-16
footers/footer.tsx:20-50		 Hard-coded English strings ("404 - Not Found", "Powered by:", "Terms of Service", etc.) with no i18n keys, plus three footer links pointing to href="#". The t('key') || 'English' pattern in landing.tsx/header.tsx also rarely fires since i18next returns the key. Use useTranslation keys with i18next defaultValue; give footer links real hrefs.
Quality course-socket course-socket.ts:38,51,65 console.log statements left in production socket code, including logging message content at line 51. Remove or gate behind config.DEVELOPMENT.

Test Coverage

9 gaps

Well Covered

  • api-client.ts — reactive 401 refresh, single-flight dedupe (concurrent + sequential races), proactive-refresh triggers, network/parse errors
  • external-api-client.ts — all three token tiers, absolute-URL passthrough, 401 retry, 204, error paths
  • exchange-token.tsx loader — missing-token redirect, registration vs registered branching, error bubbling
  • courses api + components + queries (cards, info, list, view, credential, certificates)
  • LTI: form, iframe, launch-view, launch params; course-socket connect/keep-alive/reconnect-on-close
  • self-registration hooks/api/queries/components; combobox (incl. aria fallbacks)
  • app shell: provider, app/index, router, error-fallback, layout-wrapper
  • hooks: use-maximize-window, use-feature-flags, use-translation; runtime-config, datadog router selection

Missing / Thin

  • No tests for any Zustand store (auth/course/app/registration) — the getAccessToken expiry side effect, isAuthenticated logic, and getRegistrationMetadata defaulting are all untested
  • sanitizeHtml has no XSS/bypass test — the security-critical function is effectively untested (migrate to DOMPurify, then test javascript: hrefs / nested SVG)
  • course-socket.test.ts actually asserts postMessage(..., '*') — locking in the wildcard; update once origin is fixed
  • course-socket: no test for license-change (stale-socket bug), unmount cleanup/disconnect, or reconnect backoff/cap
  • datadog: no URL/token scrubbing test (because no beforeSend exists)
  • launch-darkly.ts/provider: getLDConfig (null when no client id) and buildContext untested
  • router.tsx: the router.subscribe re-subscription bug isn't covered (a new-queryClient test would catch it)
  • prefetch-startup-data loader and page-meta instance switching (llp vs rqi) untested
  • credential-email-form.tsx, parse-optional-id.ts, utils/format.ts have no dedicated tests
