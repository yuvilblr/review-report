⚠ Verdict: Needs changes before next release. Codebase is well-structured with strong architecture (Bulletproof React layout, TanStack Query, Zustand stores). Three issues need attention: an XSS vector via a home-grown sanitizer, two postMessage calls with wildcard target origin, and a token-refresh race that leaks soon-to-expire tokens into in-flight requests.

Must Fix

3 issues
Severity Module Location Issue Suggested Fix
Critical courses course-info.tsx:35-58 course.longDesc and launchDesc are rendered via dangerouslySetInnerHTML after passing through a home-grown sanitizeHtml() that only strips <script/iframe/embed/object> tags and on* attributes. It misses javascript: URLs in href, data: URLs in src, <style> blocks, SVG xlink:href JS execution, and CSS expression attacks. The code's own comment says: "For production with untrusted content, use DOMPurify library." Replace with DOMPurify.sanitize(html). Add tightly-scoped allowlist for tags/attrs needed by course HTML (e.g. p, strong, em, a, ul/li). Strip href values that don't start with https?:.
Critical courses / socket course-socket.ts:56 — courses-view.tsx:46 window.opener?.postMessage('cdp_course_completed', '*') uses a wildcard target origin. Any page that opened the LP UI in a popup can intercept this completion event by listening to message events. While the payload is just a string, an attacker can use this as a signal that a user finished a specific course, leak that to an analytics endpoint, or fire phishing follow-ups timed to completion. Pass the LMS origin as the second arg: window.opener.postMessage('cdp_course_completed', LMS_ORIGIN). The opener origin should be configured in env (e.g. VITE_LMS_ORIGIN) or derived from document.referrer at first launch and stored in sessionStorage.
Critical api-client api-client.ts:51-56 getValidToken() kicks off refreshAccessToken() as a fire-and-forget promise (.catch(() => {})) and returns the soon-to-expire token to the current request. The request can race with the actual expiry — server sees expired JWT, returns 401, client retries via the 401 handler, but the original logical operation may have side effects already (e.g. cache writes triggered by a stale 200 response that the user-side think succeeded). Make getValidToken async: await refreshAccessToken() when expiring soon, then return the fresh token. Since refreshAccessToken already deduplicates concurrent calls via refreshPromise, parallel requests share one refresh round-trip.

Security Findings

2 issues
Severity Module Location Issue Suggested Fix
Security api-client api-client.ts No CSRF token is sent on state-changing requests (POST /auth/refresh, etc). The flow relies entirely on cookie SameSite mode for CSRF protection. If the backend ever sets the refresh cookie as SameSite=None (e.g. for cross-domain embedding scenarios), a malicious site could trigger a refresh on a victim's behalf and observe the response. Defence-in-depth is missing. Add a CSRF token round-trip: backend sets a non-HttpOnly csrf_token cookie on auth, frontend reads it via document.cookie and sends it as X-CSRF-Token header on state-changing requests. Backend validates cookie === header (double-submit pattern). Or rely on credentials: 'include' only when origin is whitelisted server-side.
Security routing app/router.tsx No client-side auth guard (e.g. RequireAuth wrapper) on protected routes. The only isAuthenticated() check is in self-registration/index.tsx. Other routes (courses, lti, certificates) mount and fire queries before realising the user has no token; if the user lands directly on a protected route without auth, they see error fallbacks and unnecessary 401-then-refresh round-trips. Not a security issue (server still enforces), but degraded UX. Add a <RequireAuth> wrapper component that checks useAuthStore and redirects to /auth when missing. Wrap protected route definitions consistently. Keeps server-side auth as the source of truth but makes client behaviour predictable.

Should Fix

9 issues
Severity Module Location Issue Suggested Fix
Resilience course-socket course-socket.ts:34-77 On reconnect, openSocket() creates a new WebSocket and assigns onopen/onmessage/onclose/onerror handlers. The previous socket is dropped via socket = ws without explicit removeAllListeners/close(). Since handlers are property assignments (not addEventListener), reassignment overwrites — but if openSocket is called twice quickly (race between onclose reconnect timer and an external connect() call), the previous socket reference is leaked. At top of openSocket(), close the previous socket if it exists: if (socket) { socket.onclose = null; socket.close(); }. Also clear any pending reconnectTimer before scheduling a new one.
Resilience course-socket course-socket.ts:68 reconnectTimer = setInterval(openSocket, RECONNECT_MS) uses setInterval, which keeps firing every 5s. Since openSocket guards via readyState check, repeat calls are no-ops once connected — but the interval is only cleared inside onopen. A failed reconnect attempt (e.g. CORS, DNS) leaves the interval firing every 5s indefinitely. Use setTimeout for the reconnect: schedule one attempt, let onclose reschedule on subsequent failures. Cap with a max retry count or exponential backoff to avoid infinite hot loops on permanent failures.
Resilience course-socket course-socket.ts:39, 50, 65 console.log('[CourseSocket] ...') debug statements remain in production code. Each connect/message/disconnect logs to the user's browser console. Looks unprofessional, leaks internal state names, and bloats console output for debugging real issues. Wrap with if (isDevelopment) from @/lib/constants, or remove. The datadogRum.addAction calls already capture these events for observability.
Type Safety api-client api-client.ts:90 apiClient<T = any> defaults the generic to any, which silently accepts whatever shape comes back. Most call sites pass a type, but a forgotten one types the response as any with zero compile-time checking. Default to unknown instead of any. Forces every caller to pass a generic or perform a runtime type check (Zod), catching contract drift earlier.
Type Safety api-client api-client.ts:135-138 catch (error) throws an object literal { message, status, code, traceId } not extending Error. Stack traces are lost; error instanceof Error checks downstream return false; React Error Boundary fallbacks may not display them correctly. Define a custom ApiError extends Error class. Throw instances of it. Preserves stack, plays nicely with logging tools, and consumer instanceof checks work.
Performance react-query react-query.ts:1-9 Global config sets staleTime: 60_000 and refetchOnWindowFocus: false for ALL queries. License/courses/announcements data has different freshness needs. A user switching tabs and coming back doesn't see updated announcements; user-license data may be stale longer than wanted on critical paths. Keep a sensible default but override per-query in queryOptions: announcements staleTime: 0, refetchOnWindowFocus: true; license/curriculum: longer staleTime as appropriate.
Storage i18n i18n/config.ts:22-91 Five separate localStorage keys (supportedLanguages, languageNames, orgDefaultLanguage, preferredLanguage, etc.) — non-atomic updates can leave the i18n state inconsistent if writes are interrupted. Also no quota-exceeded handling. Consolidate into a single i18n_config JSON object stored under one key. Wrap reads/writes in try/catch for quota errors, fall back to defaults.
Concurrency stores use-auth-store.ts:91-95 getAccessToken() performs a side effect (set({ ... }) when token is expired) inside a getter. Selectors that subscribe to accessToken will trigger an extra render cycle the moment a getter is called and detects expiry, even if no setter explicitly called. Move the expiry-clear logic out of the getter into an explicit checkExpiry() action invoked at startup and on token-aware navigation. Getters should be pure reads.
UX errors error-fallback.tsx:127 console.error('Error caught:', errorDetails) in the error fallback — but no datadogRum.addError integration, so user-facing errors don't make it to the observability pipeline unless the RUM SDK auto-captures the React error. Add datadogRum.addError(error, errorDetails) in the fallback so React errors are correlated with the user's session in RUM, and tagged with route/license context.
