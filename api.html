After re-triaging to a strict bar (real harm in normal operation or realistically exploitable), there is one conditional must-fix (LIS callback idempotency) and a handful of real security issues — a fail-open JWT secret fallback (High), a launch-token single-use race (Moderate), and a spoofable trust proxy (Moderate). Most items previously tagged critical/security were config toggles, header hardening, or defense-in-depth and are now Should Fix.

High license / LIS lis-completion.service.ts:233-253 processLisCompletion has no completion-state guard and calls the LIS service's own processCertificateClaim (line 35), which invokes msService.createCertificate with no dedup cache (unlike certificates.service.ts, which dedups by attemptId). A callback endpoint should be idempotent. Conditional: duplicate eCards occur only if the completion callback is delivered more than once and MS doesn't dedupe — but LIS/LTI callbacks are commonly at-least-once. Guard on a persisted completion flag (status/timestamp or dedupe on attemptId) before claiming; return "Already Processed" on repeats. Verify whether MS already dedupes — if so this drops to Should Fix.

Security Findings

4 issues
SeverityModuleLocationIssueSuggested Fix
High auth jwt.strategy.ts:19 secretOrKey: configService.get('auth.jwt.secret') ?? 'your-secret-key'. The consequence is severe (forged tokens / total auth bypass) if the secret is ever unset — a silent fail-open to a publicly-known value. Not exploitable when the secret is set in every env (it should be), so this is "remove the dangerous default," not an active hole. Remove the ?? 'your-secret-key' fallback and fail fast at boot when the secret is absent.
Moderate auth token.service.ts:531-552 Launch-token single-use is a check-then-act race (exists then set) — two requests fired in the same instant can both redeem one token. Narrow window; sequential replay is correctly blocked. Separately, the 'used' marker TTL is a flat 60s while JWT_LAUNCH_TOKEN_EXPIRATION is unset — if the token outlives 60s it becomes replayable after a minute (no race needed). Use atomic SET … NX for the claim, set the marker TTL to the token's remaining lifetime, and ensure JWT_LAUNCH_TOKEN_EXPIRATION is short.
Moderate bootstrap main.ts:27 expressApp.set('trust proxy', true) trusts the whole X-Forwarded-For chain, so req.ip is spoofable — weakening per-IP rate limiting and the maintenance VPN/IP allowlist. Set a specific proxy-hop count or a CIDR allowlist matching the ingress topology.
Low dispatch dispatch.service.ts:215 The Valkey cache key embeds the raw vendor secret (vendor:${vendorId}:${vendorKey}), exposing it via KEYS/SCAN and Redis logs. Requires Redis/log access to exploit. Hash the key (sha256(vendorKey)) or key only on vendorId.

Should Fix

SeverityModuleLocationIssueSuggested Fix
Config config / CORS config/app.config.ts:8 corsOrigins is hardcoded [] and never reads CORS_ORIGINS, so the non-dispatch CORS allowlist is always empty. Fail-closed config. Promote to Must Fix if the SPA calls the API cross-origin with cookies (then it blocks auth). Parse process.env.CORS_ORIGINS (comma-split), or remove the unused branch.
Config config / throttle throttle.config.ts:4 Rate limiting is off unless THROTTLE_ENABLED=true. This is a deployment toggle (set it on in deployed envs), not a code vulnerability — flagged so the default doesn't surprise anyone. Default to enabled (!== 'false') and set THROTTLE_ENABLED=true in deploy configs.
Lifecycle cache cache.module.ts:13-25 No onModuleDestroy/quit() on the Valkey client. Not a leak (the OS reclaims the socket on process exit); this is graceful-shutdown hygiene so the SIGTERM window can close Redis cleanly, and so the expiry job doesn't depend on process.exit(). Add a disposer that await client.quit() in onModuleDestroy.
Correctness rqillp-service rqillp.service.ts:448-458 updateUserCustomProfile doesn't await the PATCH before the two floating cacheService.delete calls — a concurrent read can re-cache stale data, and only two hardcoded keys are busted. Bounded (TTL-limited staleness; CacheService won't throw). await the request, then the deletes; invalidate by key pattern.
Security bootstrap main.ts No Helmet / security-response headers (helmet isn't a dependency). Header hardening, not an active vuln. Add helmet() tuned for the LTI iframe flows.
Config config / auth auth.config.ts:36-39 Refresh cookie sameSite defaults to 'none'; secure is forced only in production, so a non-prod deploy could emit an insecure cross-site cookie. Default sameSite to 'lax' and/or force secure outside local dev.
Performance axios axios.service.ts:9-42 No keep-alive agent on the shared axios instance, so every outbound call re-handshakes TCP+TLS; interceptors are no-ops. Add keepAlive: true agents; remove/implement the interceptors.
Performance auth / integrations external-auth.service.ts:57-290 No single-flight coalescing on token fetches — a cold cache lets N concurrent requests all hit the token endpoint (stampede). Coalesce in-flight fetches via a per-key promise map.
Correctness common / http http-error.handler.ts:20-29 Upstream 403→401 and 500→503 rewrites lose the real status/semantics. Map 403→ForbiddenException, 500→InternalServerError/BadGateway.
Correctness dispatch dispatch.service.ts:566-589 Check-then-act race in getOrCreateUserLicense; the partial unique index makes a concurrent second create throw P2002, which is not caught → unhandled 500. Catch P2002 and re-query/return the existing row.
Resilience dispatch dispatch.service.ts:719-736 Existing-user branch of resolveLicenseAndUserId doesn't wrap findOne in try/catch (the self-reg branch does) → a DB blip throws 500 instead of degrading to incomplete. Wrap it in the same try/catch returning nulls.
Error Handling auth auth.service.ts:498-519 exchangeExternalServiceToken swallows the error and always returns ServiceUnavailableException (even for 401s); userToken! is non-null asserted though optional. Map auth failures to UnauthorizedException; guard the missing token.
Correctness curriculum dto/course-launch-request.dto.ts:51-66 customDueDate/customStartDate/customCourseName are declared (two as required) but launchCourse() never reads them — clients must send ignored fields. Consume them in the LTI payload, or remove (at least make optional).
Config config config/app.config.ts:9-11 rqillpServiceUrl defaults to a hardcoded dev URL — a missing env var silently routes core LMS traffic to dev. Require the env var; fail fast if unset.
Robustness bootstrap main.ts:39-46 The global text/plain→urlencoded re-parse runs on every such request with no body-size guard and a fragile heuristic; req.body is reassigned mid-pipeline. Scope it to the LTI/dispatch routes and add a size limit.
Config config / auth auth.config.ts:8-12 JWT lifetimes (access/refresh/launch/registration) have no fallback — if env vars are missing they're undefined (often "never expires"). Provide explicit safe defaults or validate presence at startup.
